# 2. 系统架构

## 2.1 整体架构

Rhizome 采用分层架构设计，从上到下依次为：用户界面层、API 服务层、Agent 核心层、知识管理层和存储层。

## 2.2 核心组件

### 2.2.1 用户界面层 (UI Layer)

基于 Vue 3 + Vite 构建的单页应用，提供：

- **对话界面**：实时对话交互，支持流式输出
- **知识目录视图**：树状结构展示知识体系
- **统计信息面板**：展示知识库规模和使用情况
- **搜索功能**：快速检索已有知识
- **Markdown 渲染**：支持代码高亮、表格、列表等格式

### 2.2.2 API 服务层 (API Layer)

基于 FastAPI 构建的 RESTful API 服务：

- **Chat API**：对话接口，支持普通和流式响应（SSE）
- **Knowledge API**：知识管理接口，CRUD 操作
- **Catalog API**：目录管理接口，树形结构操作
- **依赖注入**：统一的状态管理，单例模式

### 2.2.3 Agent 核心层 (Agent Layer)

**QAAgent** 是系统的核心 orchestrator，负责：

1. **问题理解**：调用 LLM 分析问题，提取关键词和意图
2. **知识检索**：在知识库中查找相关信息
3. **答案生成**：结合上下文生成回答
4. **知识整合**：将新知识添加到知识体系

### 2.2.4 知识管理层 (Knowledge Management Layer)

**CatalogManager** 负责知识目录的维护：

- 创建、更新、删除知识目录
- 维护目录的层级关系
- 匹配问题与最佳目录

**KnowledgeStore** 负责知识的存取：

- 添加、更新、删除知识条目
- 向量化存储和检索
- 相似度搜索

### 2.2.5 存储层 (Storage Layer)

**JSON Storage**：

- 存储知识目录结构 (catalog.json)
- 存储知识条目元数据 (knowledge.json)
- 轻量级，无需额外服务

**Vector Store (Chroma)**：

- 存储知识向量嵌入
- 支持相似度检索
- 本地持久化

## 2.3 数据流

### 2.3.1 用户提问流程

```
用户提问
    │
    ▼
┌─────────────────┐
│  相似问题检查    │ ──yes──▶ 返回已有答案
└────────┬────────┘
         │no
         ▼
┌─────────────────┐
│  问题分析       │ ──▶ 提取关键词、意图、领域
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  目录匹配       │ ──▶ 找到最佳匹配目录或创建新目录
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  知识检索       │ ──▶ 在匹配目录下检索相关知识
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  生成答案       │ ──▶ 结合上下文和知识生成回答
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  更新知识库     │ ──▶ 保存新知识到目录和向量库
└─────────────────┘
```

### 2.3.2 检索优化策略

当用户提问时，系统采用**目录优先**的检索策略：

1. **目录匹配**：将问题与已有目录关键词进行匹配
2. **目录树展开**：获取匹配目录及其所有子目录
3. **范围限定检索**：只在展开的目录范围内进行向量检索
4. **结果排序**：按相似度排序返回结果

这种策略的优势：

- **高效性**：只需扫描相关目录，避免全库扫描
- **准确性**：限定范围提高检索相关性
- **可扩展性**：知识增长不影响检索效率

```

    ├── json_storage.py     # 依赖: models
    └── vector_store.py     # 依赖: config (embedding)
```

## 2.5 扩展性设计

系统设计考虑了多方面的扩展性：

### 2.5.1 LLM 扩展

添加新的 LLM 供应商，只需：

1. 在 `qa_agent.py` 的 `create_llm()` 函数中添加新的分支
2. 实现对应的 ChatModel 接口
3. 在 `.env` 中配置新的 `LLM_PROVIDER`

### 2.5.2 存储后端扩展

替换存储后端（如使用 PostgreSQL 代替 JSON），只需：

1. 实现与 `json_storage.py` 相同的接口
2. 修改 `catalog_manager.py` 和 `knowledge_store.py` 的初始化逻辑

### 2.5.3 UI 扩展

替换 Gradio 为其他前端框架，只需：

1. 实现相同的 Agent 调用接口
2. 保持 JSON 存储和向量存储不变
